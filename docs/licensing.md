# Licensing Strategy (W11)

本文档定义 Circuit Social 在平台化阶段的许可证策略与合规边界，用于指导社区版与商业版的发布。

## 目标

- 保护核心品牌与商业化能力，同时保持社区协作效率。
- 明确“可开源范围”与“商业授权范围”，降低二次分发争议。
- 为后续 SaaS 托管版与企业私有化部署提供统一法务基础。

## 版本与许可证

- `Community Edition (CE)`:
  - 建议许可证：`AGPL-3.0-only`
  - 适用范围：基础身份与社交能力、公开 API、SDK 基础模块、最小可用管理能力
  - 义务：网络分发场景下，修改版需开源并保留许可证声明
- `Commercial Edition (EE)`:
  - 许可证：商业授权协议（License Key + 合同）
  - 适用范围：企业级插件能力（VC、审计、SLA、高级安全策略、企业支持）
  - 分发方式：二进制或私有仓库，不对外开放源代码

## 模块边界原则

- CE 与 EE 必须通过稳定接口集成，不允许 EE 反向侵入 CE 核心域模型。
- CE 中保留插件扩展点定义（接口 + 生命周期），EE 仅实现扩展点。
- 对外 API 保持兼容优先，EE 增强能力通过可选字段或可选端点暴露。

## 开源范围（Community Edition）

- `apps/web`：基础登录、聊天、通讯录、朋友圈、最小配置页
- `apps/api`：认证、会话、社交、DID、基础租户应用配置
- `packages/sdk-js`、`packages/sdk-react`：公开 SDK 与基础示例
- `docs/*`：公开架构、错误码、测试与运维文档（不含商业实施细节）

## 商业能力范围（Commercial Edition）

- 企业可验证凭证（VC）签发与策略编排
- 审计增强：长期留存、检索、导出与合规报表
- SLA 能力：限流策略模板、专属告警、优先支持通道
- 租户高级能力：品牌化主题、细粒度 RBAC、Key 托管策略

## 合规与发布要求

- 每次发布需校验第三方依赖许可证，不得引入与 AGPL/商业协议冲突的依赖。
- CE 发布包必须包含：LICENSE、NOTICE、变更说明。
- EE 发布包必须包含：授权协议版本、支持条款版本、到期策略说明。
- 对外文档中不得将 EE 专有实现细节写入 CE 仓库。

## 贡献与 CLA

- 建议启用 Contributor License Agreement（CLA）或 DCO 流程。
- 社区贡献默认进入 CE；若涉及 EE 需求，需先抽象为通用扩展点再合并。

## 待法务确认事项

- AGPL 是否调整为 SSPL 或双许可证（AGPL + 商业）模式。
- 商业授权最小计费单元（租户 / MAU / API 调用量）。
- 私有化部署附带源码可见范围与升级条款。
